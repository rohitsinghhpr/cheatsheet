Here are some common terms used in manual testing, which is the process of testing software manually by executing test cases without the help of automation tools:

1. Test Case

Definition: A test case is a specific set of actions or conditions used to verify a particular aspect of the software. It includes a set of input conditions, execution steps, and expected results.

Example: A test case for logging into an application could include steps like entering the username and password, then verifying whether the user is redirected to the homepage upon successful login.

2. Test Plan

Definition: A test plan is a detailed document that outlines the strategy, scope, approach, resources, schedule, and objectives of the testing effort. It provides an overview of the testing process.

Example: A test plan may specify the testing objectives, such as validating the functionality of a shopping cart, and define the types of testing (functional, regression, etc.) to be conducted.

3. Test Suite

Definition: A test suite is a collection of related test cases grouped together to test a particular feature or functionality of the software.

Example: A test suite for an e-commerce website might include test cases for user registration, login, browsing products, adding to the cart, and making payments.

4. Bug/Defect

Definition: A bug (or defect) is an issue or error found in the software that causes it to behave unexpectedly or incorrectly. Bugs are typically documented and tracked for resolution.

Example: A bug could occur when a "Submit" button on a form doesnâ€™t trigger the expected action, like submitting data.

5. Regression Testing

Definition: Regression testing involves re-running previously executed test cases to ensure that new changes, such as bug fixes or feature additions, have not introduced new issues or affected existing functionality.

Example: After fixing a bug in the login functionality, regression testing would verify that the login, as well as other areas of the software, still works correctly.

6. Smoke Testing

Definition: Smoke testing, also known as "build verification testing," is a preliminary testing phase to check if the major functions of the software work. It ensures the software build is stable enough for further testing.

Example: After receiving a new build, a tester might verify that the application starts and key features like login and basic navigation are functioning before proceeding to more detailed testing.

7. Sanity Testing

Definition: Sanity testing is a quick, shallow testing process used to verify that specific functionality or bug fixes are working as expected. It is narrower in scope than full regression testing.

Example: After a bug fix is applied to the login feature, sanity testing would check if the fix works and that the login functionality behaves correctly.

8. Ad-hoc Testing

Definition: Ad-hoc testing is an informal type of testing where the tester explores the application without any predefined test cases. The aim is to discover defects by randomly testing the application.

Example: A tester might click through various sections of a web app to see if they can find any unexpected issues or crashes.

9. Exploratory Testing

Definition: Exploratory testing is an unscripted, simultaneous learning and testing process, where testers actively explore the application to understand its functionality and look for defects, while adapting their testing approach based on findings.

Example: A tester might test a new feature by using it as an end-user would, without predefined test cases, while simultaneously learning how the feature works.

10. User Acceptance Testing (UAT)

Definition: User Acceptance Testing (UAT) is the final testing phase where actual users validate the software to ensure it meets their needs and requirements before release.

Example: In an e-commerce application, UAT could involve users checking if they can complete tasks like browsing products, adding them to the cart, and completing a purchase.

11. Test Environment

Definition: A test environment is a setup of hardware, software, and network configurations where the testing is executed. It mimics the production environment.

Example: For testing a mobile app, the test environment might include specific mobile devices, OS versions, and network conditions to simulate real-world usage.

12. Defect Life Cycle (Bug Life Cycle)

Definition: The defect life cycle represents the journey of a bug from detection to resolution. It defines the different states a defect goes through during the testing process.

Example: A defect might start in the "New" state, move to "Assigned" to a developer, change to "Fixed," and then undergo "Retesting" before being marked as "Closed" if resolved successfully.

13. Boundary Value Analysis

Definition: Boundary Value Analysis (BVA) is a testing technique that focuses on testing the boundaries of input data values. It is based on the premise that errors often occur at the boundaries rather than in the middle of the input range.

Example: For a form field that accepts numbers from 1 to 100, boundary value testing would include values like 0, 1, 100, and 101 to test how the system handles inputs at the edges of the range.

14. Test Report

Definition: A test report is a document that provides detailed information about the testing process, including the test cases executed, the results, defects found, and the overall status of testing.

Example: After completing the testing of a software release, a test report may summarize which features were tested, which bugs were found, and whether the software is ready for release.

15. Priority and Severity

Definition:

  • Priority: Refers to how quickly a bug should be fixed based on business impact and urgency.
  • Severity: Refers to the seriousness of the bug and its impact on the functionality of the software.

Example:

  • A bug that causes the application to crash might be high severity, but if it only happens during a rare scenario, it might be of low priority.
  • A typo in the footer of a website might have low severity but high priority if it affects a major marketing campaign.

16. Test Execution

Definition: Test execution is the process of running the test cases on the software to validate its functionality.

Example: A tester manually follows the steps outlined in the test cases, inputs the specified data, and records the actual outcome of the test to compare with the expected results.

17. Test Data

Definition: Test data refers to the data used during test execution to evaluate the behavior of the software. It can be valid or invalid, depending on what is being tested.

Example: For a login feature, valid test data might include correct username and password, while invalid test data could be incorrect credentials.

These terms represent just a fraction of the terminology in manual testing, but they cover some of the most important aspects of the process used to validate the functionality, quality, and reliability of software applications.
